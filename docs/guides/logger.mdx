This guide explains how Fentaris logging works and how to configure it in depth. You will learn every configurable parameter, see examples, and get practical tips for production.

If you only need the high-level concept, see [Core logger](/core/logger).

## Overview

Fentaris uses a structured `Logger` with a pluggable `LoggerDriver`. The logger is available in middleware, routes, and events as `ctx.log` and is also used internally by the proxy.

At a minimum you can instantiate a default logger:

```ts theme={null}
import { Logger } from "@fentaris/core";

const logger = new Logger();
```

Then pass it to the proxy:

```ts theme={null}
const proxy = new McpProxy({
  servers: [filesystem],
  logger,
});
```

## Logger parameters

The `Logger` constructor accepts an options object. The most important parameter is the `driver`.

### driver

`driver` defines where log entries are written. It must implement a `write(entry)` method.

```ts theme={null}
const logger = new Logger({
  driver: {
    write(entry) {
      console.info(entry.message, entry.metadata);
    },
  },
});
```

**Tip:** If you already use a log aggregator, forward `entry` to it inside `write`.

## Logger entry shape

The logger emits structured entries with a consistent shape. Use this structure for querying and analytics.

```ts theme={null}
type LogEntry = {
  level: "debug" | "info" | "warn" | "error";
  message: string;
  metadata?: Record<string, unknown>;
  tags?: Record<string, string>;
  timestamp: string;
};
```

## Logging levels

Use these levels consistently:

- `debug`: noisy, development or diagnostics only.
- `info`: normal operations like tool calls and routing.
- `warn`: recoverable issues or policy denies.
- `error`: failures that need attention.

```ts theme={null}
ctx.log.debug("tool.input", { tool: ctx.tool?.proxyName });
ctx.log.info("tool.call", { tool: ctx.tool?.proxyName });
ctx.log.warn("tool.deny", { reason: "policy" });
ctx.log.error("tool.error", { message: "upstream failed" });
```

## Tags and metadata

Tags are indexed keys you want on every log line, like `traceId` or `tenantId`. Metadata is per-entry detail.

```ts theme={null}
proxy.use(async (ctx, next) => {
  ctx.log.setTag("trace", ctx.user.traceId ?? "no-trace");
  ctx.log.setTag("tenant", String(ctx.user.tenantId ?? "unknown"));

  ctx.log.info("tool.call");

  return next();
});
```

**Tip:** Keep tags short and stable. They should be consistent across all events.

## Custom driver examples

### JSON log lines

```ts theme={null}
const logger = new Logger({
  driver: {
    write(entry) {
      process.stdout.write(`${JSON.stringify(entry)}\n`);
    },
  },
});
```

### Forward to a logging service

```ts theme={null}
const logger = new Logger({
  driver: {
    write(entry) {
      sendToLogPipeline({
        level: entry.level,
        message: entry.message,
        tags: entry.tags,
        metadata: entry.metadata,
        ts: entry.timestamp,
      });
    },
  },
});
```

### Redis logger driver

Use `RedisLoggerDriver` when you want Fentaris to append structured log entries to a Redis list.

```ts theme={null}
import { Logger, RedisLoggerDriver } from "@fentaris/core";

const logger = new Logger({
  driver: new RedisLoggerDriver({
    client: redis,
    key: "fentaris:logs",
  }),
});
```

## Auto logging

Enable proxy auto logging to record tool request, response timing, user context, and policy metadata.

```ts theme={null}
const proxy = new McpProxy({
  servers: [filesystem],
  logger,
  autoLog: true,
});
```

## Logger in middleware and hooks

The same logger instance is exposed as `ctx.log`. Fentaris enriches it with safe metadata such as operation, subject id, server name, tool name, proxied tool name, resource URI, prompt name, completion target, transport type, session id, credential source, and policy outcome where available.

```ts theme={null}
proxy.on("tool:success", async ({ ctx, durationMs }) => {
  ctx.log.info("tool.success", { durationMs });
});
```

Resource, prompt, and completion operations also emit lifecycle events:

```ts theme={null}
proxy.on("resource:success", async ({ ctx, durationMs }) => {
  ctx.log.info("resource.success", {
    uri: ctx.resource?.uri,
    durationMs,
  });
});
```

## Error handling and logging

When you catch errors in middleware, log them with a stable message and include safe metadata.

```ts theme={null}
proxy.use(async (ctx, next) => {
  try {
    return await next();
  } catch (error) {
    ctx.log.error("tool.error", { message: String(error) });
    return ctx.deny("Tool failed. Try again later.");
  }
});
```

## Best practices

- Prefer structured fields (`tool`, `server`, `userId`) over free-form strings.
- Avoid logging secrets or raw payloads.
- Use tags for correlation (`traceId`, `tenantId`).
- Keep message names consistent (`tool.call`, `tool.deny`).
- Log denies at `warn` to make policy violations visible.
