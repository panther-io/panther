This guide shows a hardened proxy setup that you can adapt for staging and production. It focuses on repeatable patterns: naming, routing, policy enforcement, and clean shutdowns. The goal is to make your proxy predictable for clients and easy to operate for your team.

If you are new to Panther, start with the [Quickstart](/getting-started/quickstart) and come back here when you need to operationalize the proxy.

## Choose a stable endpoint

Client integrations will hard-code the MCP endpoint. Keep the `path` stable, even if ports or hosts change. If you deploy behind a reverse proxy, the public URL can stay the same while the internal proxy moves.

```ts theme={null}
const proxy = new McpProxy({
  servers: [filesystem],
  port: Number(process.env.PORT ?? 3000),
  path: "/mcp",
});
```

## Configure proxy identity

```ts theme={null}
const proxy = new McpProxy({
  servers: [filesystem, diagnostics],
  name: "panther-proxy",
  version: "0.1.0",
  port: 3000,
  path: "/mcp",
});
```

**Why it matters:** the `name` and `version` are surfaced to MCP clients during initialization. Treat these as your control plane identity so you can trace which deployment handled a given tool call.

## Add consistent server metadata

Keep names stable and use `displayName` for human-friendly labels. Stable names become the prefix for tool names, so they should remain consistent across environments.

```ts theme={null}
const analytics = new McpServer({
  name: "analytics",
  displayName: "Analytics",
  transport: analyticsTransport,
});
```

**Naming tip:** if you operate multiple environments, use the same server names in prod and staging. Avoid names like `analytics-staging` because that changes the tool prefix and breaks clients.

## Harden tool discovery

Use `tools:list:after` to transform the tool list before exposing it to clients.

```ts theme={null}
proxy.on("tools:list:after", ({ tools }) => {
  return tools?.filter((tool) => !tool.name.includes("diagnostics__experimental"));
});
```

**Tip:** Keep the allow/deny logic in one place. This makes it easier to audit which tools are visible in production versus staging.

## Add environment-aware tool lists

If you need stricter tool lists in production, filter at discovery time. This prevents tools from even showing up in the client UI.

```ts theme={null}
proxy.on("tools:list:after", ({ ctx, tools }) => {
  if (ctx.user.environment !== "prod") {
    return tools;
  }

  return tools?.filter((tool) => !tool.name.includes("__experimental"));
});
```

## Customize request logging

```ts theme={null}
proxy.use(async (ctx, next) => {
  ctx.log.info("proxy.request");

  return next();
});
```

**Tip:** add a stable request tag in the user resolver (for example `traceId`) and include it in each log entry so you can trace a single tool call across systems.

## Enforce policy at the edge

Centralized policy is the biggest benefit of a proxy. Keep durable authorization in `Policy` and `Group` declarations, then use middleware and server-scoped tool routes for runtime validation.

```ts theme={null}
proxy.tool("filesystem.delete", async (ctx, next) => {
  if (ctx.user.role !== "admin") {
    return ctx.deny("Delete is disabled for this role.");
  }

  return next();
});
```

Consider layering policies: first deny destructive calls for non-admins, then enforce plan-based access, and finally enforce per-tenant limits.

## Add graceful shutdowns

When the proxy shuts down, it should close all active transport connections. This avoids orphaned server processes and incomplete tool calls.

## Close cleanly

```ts theme={null}
process.on("SIGINT", async () => {
  await proxy.close();
  process.exit(0);
});
```

Also handle `SIGTERM` in production (Kubernetes, systemd, or PaaS) to avoid orphaned MCP server processes.

```ts theme={null}
process.on("SIGTERM", async () => {
  await proxy.close();
  process.exit(0);
});
```

## Complete Express-like proxy example

```ts theme={null}
import { McpProxy, McpServer, Policy, group, user } from "@panther/core";

const github = new McpServer({
  name: "github",
  transport: githubTransport,
});

const supportPolicy = Policy.allowAll("support");
const proxy = new McpProxy({
  servers: [github],
  groups: [
    group({
      id: "support",
      users: [user("alice")],
      policy: supportPolicy,
    }),
  ],
  autoLog: true,
});

const githubApp = proxy.server("github");

proxy.use(async (ctx, next) => {
  ctx.log.setTag("subject", ctx.subject?.id ?? "anonymous");
  return next();
});

githubApp.tool("create_issue", async (ctx, next) => {
  if (typeof ctx.args?.title !== "string") {
    return ctx.deny("Issue title is required.");
  }

  return next();
});

githubApp.on("tool:success", async ({ ctx, durationMs }) => {
  ctx.log.info("github.tool.success", { durationMs });
});

proxy.on("tools:list:after", ({ tools }) => {
  return tools?.filter((tool) => !tool.name.endsWith("__experimental"));
});
```

Define `user` resolution in the proxy config so every request is stamped with identity and trace metadata.

```ts theme={null}
const proxy = new McpProxy({
  servers: [filesystem],
  user: (req) => ({
    id: req.headers["x-user-id"] as string | undefined,
    tenantId: req.headers["x-tenant-id"],
    traceId: req.headers["x-trace-id"],
    environment: req.headers["x-env"],
  }),
});
```

This makes it trivial to audit or enforce policy without additional lookups.

## Operational checklist

- Confirm tool lists in production match what you expect.
- Tag logs with `user.id` or `tenantId` for fast auditing.
- Ensure each `McpServer` uses a stable `name` to avoid API drift.
- Avoid hard-coding secrets; inject them via `env` per server.
- Keep deployment configuration in version control.
